The PolyStatic Programming Language

What makes PolyStatic different?

PolyStatic is targeted at static typesafety and runtime modularity. The goal is to have a type system similar to those of C++ and Java, but runtime compatibility like that of Python. The intended use is programming game logic as a scripting language.

How do I learn to write PolyStatic code?

The documentation isn't ready yet because the language is still very new. There are examples, though.

What's the Hello World program?

For there to be a Hello World program, there has to be a way to output information somwhere. PolyStatic is meant to be used by other code (e.g. a game), so it depends on the caling code to provide e.g. a debugging callback function. Since there's no standardized output, there's no Hello World program. What do you want from me? There can't be one for every language!

What are the various file extensions?

Scripts that are meant to be interpreted from text are called liquids, nad have a file extension of psliquid. Compiled PolyStatic code (e.g. a libraries) are called solids, and have a file extension of pssolid. By now you may be wondering what gasses are - files with extension psgas represent interface files, and are generally provided as APIs for host programs (e.g. callbacks) or libraries written in other languages. Basically, anything that matches *.ps* is a PolyStatic-related file.

How do I compile/run it?

You have to use the liquid interpreter for now; more options will be available later (such as building to solids or running liquids live from other expplications). In the future, you will be able to run solidify to compile your liquids into solids.

What's all this crazy terminology?

PolyStatic is made of Matter:

  • Solid: Compiled PolyStatic code, aka a library. (Ruby calls them Gems)
  • Liquid: Source PolyStatic code in raw form.
  • Gas: Declarations-only file, generally used like C++ headers as a way to interface with the host/calling code via callbacks.
  • Space: Like C++ namespaces and scopes, Java packages and scopes, etc. except unified into one term.
  • Solidifier: Compiler; converts liquids into solids.
  • Melting: Extracting reflective type & name information from solids to get an interface.
  • Liquifier: Something that takes liquids and solids, merges them into one environment, and executes particular functions at its leisure. For example, an interpreter. This is often called the "host".
  • Liquification: Executing code with a liquifier.
  • Condensation: Porting code from other languages to PolyStatic or moving host implementation to PolyStatic implementation.
  • Evaporation: Converting PolyStatic code to another langauge (generally the native language of the host).
